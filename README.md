# SQL Clinical Data Warehouse (Synthea Synthetic Patient Records)

Welcome to the **Clinical Data Warehouse** repository! 🚀  
This project demonstrates the design and implementation of a modern **clinical / EHR-style data warehouse** built on **synthetic healthcare data generated by Synthea**.  
Designed as a portfolio and learning project, it highlights best practices in **data engineering**, **SQL development**, **ETL/ELT pipelines**, and **dimensional modeling**.

---

## 🏗️ Data Architecture

The data architecture for this project follows the **Medallion Architecture** approach with **Bronze**, **Silver**, and **Gold** layers.

### Bronze Layer
Stores raw healthcare data **as-is** from the source CSV files.

- Raw data ingestion from CSV files into SQL Server
- Minimal transformation (source-aligned)
- Used for traceability, validation, and reprocessing

### Silver Layer
Contains cleaned, standardized, validated, and lightly integrated clinical data.

- Data cleansing and standardization
- Normalization and formatting (dates, numbers, null handling)
- Validation of relationships between source entities
- Preservation of temporal clinical records
- Preparation for downstream analytical modeling

### Gold Layer
Contains business-ready and analytics-ready data modeled for reporting and analysis.

- Dimensional modeling
- Fact and dimension tables
- Aggregated analytical objects
- Reporting views
- Supports KPI reporting, SQL analysis, and dashboarding

## 🧪 Dataset

This project uses the **Synthea Sample Synthetic Patient Records (CSV)** dataset, which simulates an **Electronic Health Record (EHR)** environment: patients exist over time, they have clinical visits (“encounters”), and during those encounters they may receive diagnoses, measurements, prescriptions, procedures, immunizations, and other clinical events.

The dataset includes multiple CSV files representing healthcare entities such as:

- `patients`
- `encounters`
- `conditions`
- `observations`
- `medications`
- `procedures`
- `allergies`
- `careplans`
- `devices`
- `imaging_studies`
- `immunizations`
- `organizations`
- `providers`
- `payers`
- `payer_transitions`
- `supplies`

This structure makes the dataset suitable for practicing:

- multi-table integration
- data cleaning and validation
- clinical event modeling
- fact and dimension design
- medallion-based warehouse development

|--------|---------|-------------|---------------|------------|

| Silver | Clean, standardize, validate, and integrate source-aligned data | Tables | Full Load / Full Refresh | Cleansed operational model / light relational integration |

## ⚙️ Naming Conventions

### General Principles
- Use `snake_case`
- Use English for all names
- Avoid SQL reserved words
- Keep naming consistent across all layers

### Bronze Layer
Bronze tables keep source-aligned names and mirror the original source structure as closely as possible.

**Examples**
- `bronze.patients`
- `bronze.encounters`
- `bronze.conditions`

### Silver Layer
Silver tables remain close to the source structure but contain cleaned, standardized, and validated data.

**Examples**
- `silver.patients`
- `silver.encounters`
- `silver.observations`

### Gold Layer
Gold tables use business-oriented naming conventions.

**Examples**
- `dim_patient`
- `fact_encounter`
- `fact_observation`
- `agg_monthly_encounters`
- `vw_patient_summary`

### Keys
- Source identifiers use `_id`
- Surrogate keys use `_key`

**Examples**
- `patient_id`
- `patient_key`
- `encounter_id`
- `date_key`

### Technical Columns
Technical metadata columns use a consistent prefix such as `dwh_`.

**Examples**
- `dwh_load_date`
- `dwh_batch_id`
- `dwh_source_file`

## 🔄 Load Strategy

For the scope of this project, the warehouse uses a **full-load approach**.

- **Bronze:** Full load from raw CSV files
- **Silver:** Full refresh from Bronze
- **Gold:** Full refresh from Silver

This approach is appropriate because the project is based on a **dataset snapshot**, not a live transactional healthcare system.

---

## 🕒 Scope and Historization

This project focuses on the **latest available dataset snapshot**.  
Full enterprise historization is **not implemented across the entire warehouse**; however, **temporal clinical records already present in the source data are preserved where relevant**.

This is especially important for entities such as:

- encounters
- conditions
- medications
- observations
- procedures
- payer transitions
